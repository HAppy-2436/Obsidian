# CH08 异常控制流（ECF）

> 一句话总结：**异常控制流 = 改变正常指令流的机制**，从底层硬件异常到上层的进程/信号，贯穿整章。**进程 = 运行中的程序**，进程切换靠内核；**信号 = 软件中断**，异步通知进程事件。

> 相关：[[CSAPP-CH09-虚拟内存与动态内存分配]]、[[CSAPP全部测验题目]]

---

## 1. 核心框架

```
┌─────────────────────────────────────┐
│   进程 / 上下文切换（高粒度）          │ ← shell、exec、fork、wait
├─────────────────────────────────────┤
│   信号（软中断）                      │ ← SIGINT / SIGTSTP / SIGCHLD
├─────────────────────────────────────┤
│   硬件异常（陷阱/故障/中断/终止）       │ ← 系统调用、缺页
├─────────────────────────────────────┤
│   跳转 / call / ret（普通控制流）       │ ← 不算 ECF
└─────────────────────────────────────┘
```

⚠️ **不改变控制流的机制（强制类型转换）不是 ECF**。普通函数调用、跳转是基础控制流。

📝 **真题**：以下**不属于** ECF 的 = **强制类型转换**（B）。

## 2. 异常四类（必考）

| 类别 | 同步性 | 行为 | 返回行为 |
|------|--------|------|----------|
| **中断（Interrupt）** | **异步** | I/O 设备信号 | 总是返回下一条 |
| **陷阱（Trap）** | 同步 | 系统调用、有意触发 | 返回下一条 |
| **故障（Fault）** | 同步 | 可恢复错误（如缺页） | 可能返回当前指令（重试）或终止 |
| **终止（Abort）** | 同步 | 不可恢复错误 | **不返回** |

⚠️ **陷阱 ≠ 致命错误**：陷阱是**有意触发**（如 `int 0x80` 做 syscall）；中断是**异步的**（硬件信号）。

📝 **真题**：
- A 错：中断是**异步**的
- B 错：陷阱是有意触发，**不是致命错误**
- C 错：故障处理完后**回到当前指令**（不是下一条），如果是缺页会重试
- D 对：终止异常**从不返回**给应用

## 3. 进程基础

### 3.1 三大特性

| 特性 | 含义 |
|------|------|
| **独立逻辑控制流** | 好像独占 CPU（实际是分时） |
| **私有地址空间** | 好像独占内存（实际是虚拟内存） |
| **并行 / 并发** | 同一时段 / 交错执行 |

### 3.2 用户模式 vs 内核模式

- 普通进程初始时在**用户模式**（user mode）
- 发生异常 → 转入**内核模式**（kernel mode）
- **进程上下文切换一般发生在内核模式**（D 错在"用户模式"）

### 3.3 并发进程判定（⚠️ 必考）

两个进程的运行时段**有重叠**就是并发。

例：A(1-4), B(3-6), C(5-9), D(7-11)

| 进程对 | 时段 | 重叠？ |
|--------|------|--------|
| A-B | [1,4] ∩ [3,6] = [3,4] | ✅ |
| A-C | [1,4] ∩ [5,9] | ❌ |
| A-D | [1,4] ∩ [7,11] | ❌ |
| B-C | [3,6] ∩ [5,9] = [5,6] | ✅ |
| B-D | [3,6] ∩ [7,11] | ❌ |
| C-D | [5,9] ∩ [7,11] = [7,9] | ✅ |

📝 **真题**：并发进程对 = **3 对**（A-B, B-C, C-D）（B）。

## 4. fork / exec / wait

### 4.1 fork（克隆）

```c
pid_t pid = fork();
if (pid == 0) { /* 子进程 */ }
else { /* 父进程，pid 是子进程 PID */ }
```

⚠️ **fork 一次返回两次**：子进程得 0，父进程得子 PID。

### 4.2 fork 输出题（⚠️ 高频易错）

```c
fork();
if (fork() == 0) { fork(); printf("2"); exit(0); }
printf("2");
```

📝 **真题**：输出多少行 "2"？

画进程树：

```
         P0
        /  \
     fork1  fork1  (P0 和 P1)
      |       \
   if(fork()==0) if(fork()==0)
     P0继续     P2
       |        |
   else 打印2  fork2 → P3, P4 都打印
   exit
```

让我重新画：

```
                 P0
              fork1() → P0', P1
              P0' = if(fork()==0) 分支
              fork2() → P0', P2 (P2 在 if 里)
              P0' 走 else: 打印 2
              P2 在 if 里: fork3() → P2, P3 (P3 走 fork3 后续)
              P2 打印 2, exit
              P3 打印 2, exit
              P1 走 else: 打印 2, exit
```

实际是 4 个进程打印 2（P0' / P2 / P3 / P1），**输出 4 行 2**（B）。

⚠️ **容易选错**：fork 之后每个进程都独立执行，所有 fork 出去的进程都会到达最后的 `printf("2")`，无论进入 if 还是 else。

### 4.3 execve

```c
int execve(const char *filename, char *const argv[], char *const envp[]);
```

- **不创建新进程**，直接覆盖当前进程的地址空间
- 一旦成功，**不返回**；失败返回 -1

### 4.4 shell 执行流程

用户输入命令 → shell **fork 子进程 → 子进程调用 execve**。

📝 **真题**：shell 主流程 = **fork → 子进程 execve**（C）。

### 4.5 wait / waitpid

| 函数 | 行为 |
|------|------|
| `wait(&status)` | 阻塞，直到**任意一个**子进程终止 |
| `waitpid(pid, &status, options)` | 等待**指定**子进程 |

⚠️ **wait 阻塞父进程**，不会继续运行直到子进程结束。

📝 **真题**：
- `wait(status)` ≡ `waitpid(-1, status, 0)` ✅
- wait **阻塞**父进程（B 错）——题说"调用进程继续运行"

### 4.6 子进程回收与僵尸

- 子进程终止后变成**僵尸（zombie）**：已结束但未被父进程 wait
- 必须由父进程 wait 才能彻底释放
- 如果父进程先终止，子进程变**孤儿（orphan）**，由 **init 进程**（PID=1）收养

📝 **真题**（kill 子进程场景）：
- 父 PID 6639 / 子 PID 6640 都死循环
- 执行 `kill 6640`（SIGKILL 默认），子进程被 SIGKILL → 终止
- 但 **父进程还在**，父进程没 wait → 子进程变**僵尸**（**不是孤儿**，因为父在）
- 父进程不死，子进程一直僵尸

⚠️ **孤儿 ≠ 僵尸**：
- **孤儿**：父进程没了，init 收养
- **僵尸**：子进程结束了，但父没 wait

📝 **真题选项分析**：
- A 错：父子谁先输出不确定
- B 错：子进程终止但变僵尸，**不是被回收**（回收是 wait）
- C 错：不是孤儿（父还在）
- D 错：父没死，init 没接管 → 选 C 不对 → 实际 **A/B/C/D 都错**，但只有 B 最接近"终止并被回收"的错误描述（实际是僵尸）。需要重新分析

⚠️ **这道题要选"正确的陈述"**：
- A：执行顺序**不确定**（父子都可能先输出）
- B：kill 子后，子终止 → **但父没 wait** → **僵尸**，没被回收 ❌
- C：孤儿 ❌
- D：父没死，init 不接管 ❌

实际看题目里："以下正确的陈述是"——4 个选项严格说都不全对，但**最接近的**可能要看具体语境。**B 错得最明显（"被回收"错）**。这题需要按题目上下文判断。

### 4.7 进程调度顺序

进程 A / 子 A1 / 进程 B / 子 B1 / 进程 C / 子 C1 都被挂起，重新调度的顺序：
- **不确定**，与调度器实现有关

📝 **真题**：调度顺序 = **不确定**（A）。

### 4.8 kill 命令

- `/bin/kill pid`：发 SIGTERM（可被捕获）
- `/bin/kill -9 pid`：发 SIGKILL（**不可捕获、不可忽略**）
- `/bin/kill -pid`：发信号给**进程组**
- `/bin/kill -9 -pid`：发 SIGKILL 给**进程组**

📝 **真题**：3195 是 pgrp，要 kill 整个进程组 → **`/bin/kill -9 -3195`**（C）。
- 注意 -3195 表示 PGID = 3195（负的 PID 表示进程组）

## 5. 信号（⚠️ 高频考）

### 5.1 关键信号

| 信号 | 编号 | 默认行为 | 来源 |
|------|------|----------|------|
| **SIGINT** | 2 | 终止 | Ctrl-C |
| **SIGTSTP** | 20 | 停止（挂起） | Ctrl-Z |
| **SIGQUIT** | 3 | 终止 + core dump | Ctrl-\ |
| **SIGALRM** | 14 | 终止 | alarm() |
| **SIGCHLD** | 17 | 忽略 | 子进程状态变化 |
| **SIGKILL** | 9 | 终止（不可捕获） | kill -9 |
| **SIGSEGV** | 11 | 终止 + core | 段错误 |

⚠️ **Ctrl-Z → SIGTSTP**（不是 SIGINT）。
⚠️ **Ctrl-C → SIGINT**。

📝 **真题**：Ctrl-Z 发的信号 = **SIGTSTP**（D）。

### 5.2 signal 处理

```c
signal(SIGINT, sig_handler);  // 注册 handler
```

⚠️ **signal 只是注册，不等信号**。要等信号需要 `pause()` 或 sleep()。

📝 **真题**：
```c
void sig_handler(int sig) { printf("Caught\n"); exit(0); }
int main() {
    signal(SIGINT, sig_handler);
    printf("Wait\n");
    pause();
    printf("End\n");
    return 0;
}
```

按 Ctrl-C：
1. 打印 "Wait"
2. `pause()` 阻塞
3. Ctrl-C → 内核递送 SIGINT → 跳入 handler
4. handler 打印 "Caught" → exit(0)
5. **永远不会打印 "End"**（exit 终止了进程）

输出：先 "Wait"，后 "Caught"（**2 行，D**）。

### 5.3 setjmp / longjmp（非局部跳转）

- `setjmp(env)`：**调用一次，返回一次或多次**（如果 longjmp 跳回，会再次返回）
- `longjmp(env, val)`：跳回 setjmp 处，返回 val

📝 **真题**：setjmp 的特点 = **调用一次，返回一次或多次**（D）。

## 7. 速记口诀

> **异常四类：中断异步、陷阱有意、故障可恢复重试、终止不返回**
> **fork 一次返回两次：子得 0，父得子 PID**
> **shell 执行：fork 子进程 → 子进程 execve**
> **僵尸：子结束但父没 wait；孤儿：父没了，init 收养**
> **wait 阻塞父进程，等子进程终止**
> **Ctrl-C → SIGINT；Ctrl-Z → SIGTSTP；kill -9 → SIGKILL（不可捕获）**
> **kill 进程组用 `-pid`（负号）**
> **setjmp 调用一次，返回一次或多次**
> **signal 注册不阻塞，要 pause/sleep 才等信号**
> **并发 = 时段重叠；并行 = 同时执行**
