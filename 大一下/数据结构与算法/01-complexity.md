# 时间空间复杂度入门

> [!info] 章节定位
> 本章为「复杂度分析」最小可用版本。不面面俱到讲完所有分析细节，只把后面 49 章会用到的「O 符号」「最坏情况」「时空权衡」讲清楚。完整的分析指南安排在更后面。

## 📚 学习目标

- 理解 Big O 表示法的含义，能把一段代码的复杂度估成 O(1) / O(n) / O(n²) / O(log n)
- 知道「常数项和低阶项可以省略」的简化原则
- 区分**最坏时间复杂度**与**平均时间复杂度**的使用场景
- 知道时间复杂度是「**单位时间的迭代次数**」、空间复杂度是「**额外申请的存储**」
- 能在写代码前大致预判算法会不会超时

## 🎯 一句话总结

> 复杂度是「粗略估计」而非「精确计算」：时间复杂度 ≈ 最坏情况下的循环层数；空间复杂度 ≈ 算法额外申请的存储大小；都用 Big O 表达，去掉常数和低阶项。

## 📖 正文

### 1. 为什么需要「复杂度」这个工具

写一段代码衡量它「快不快」「省不省内存」，最朴素的思路是计时器 + 内存监控。但这种方法严重依赖机器、输入数据分布、编译器优化等级，没法在面试、笔试、白板上复用。

复杂度提供了一种**与机器无关、与数据分布近似无关**的粗粒度衡量标准。掌握它，你就能在写代码之前大致判断「这个解法会不会超时 / 超内存」。

### 2. Big O 表示法的核心规则

Big O 记号是「增长率上界」的最简形式。记 `T(n)` 是算法在输入规模为 `n` 时的操作次数，则 `T(n) = O(f(n))` 意味着 `T(n)` 的增长不会超过 `f(n)` 的常数倍。

四句话记住它：

1. **只保留最高增长项**：`O(2n² + 3n + 1)` 简化为 `O(n²)`。
2. **忽略常数系数**：`O(1000n + 9999)` 简化为 `O(n)`。
3. **不区分 base**：对数无论 `log₂` 还是 `log₁₀`，都是 `O(log n)`。
4. **不写单位**：直接说 `O(n)`，不写 `O(n) operations`。

```text
常见复杂度（从快到慢）：

O(1) < O(log n) < O(n) < O(n log n) < O(n²) < O(n³) < O(2ⁿ) < O(n!)
```

### 3. 时间复杂度 = 循环层数（粗略估算）

最常用的简化判断：**算法的时间复杂度 ≈ 嵌套 for 循环的最大层数对应的复杂度**。

- 单层 for 循环遍历 n 个元素 → `O(n)`
- 两层嵌套 for 循环 → `O(n²)`
- 二分查找类「每次砍半」 → `O(log n)`
- 排序类「分治 + 合并」 → `O(n log n)`

> [!warning] 简化方法的两点不严谨
> 1. 「循环层数」只是一种粗略估计；递归、`while` 循环条件依赖输入、隐藏的内存拷贝、库函数调用等都会带来额外时间。
> 2. 大多数情况我们分析**最坏情况**复杂度；但像哈希表这种「摊销 + 概率」型数据结构，会用**平均复杂度**描述。完整方法会在后续「时空复杂度分析实用指南」中展开。

### 4. 空间复杂度 = 额外申请的存储

空间复杂度衡量算法**额外申请**了多少存储空间。注意：

- **输入数据本身不计入**。函数参数传入的 `nums` 数组、`head` 链表头，它们是题目给的，不算算法的空间消耗。
- **输出数据要计入**。如果你返回了一个和输入等大的新数组，空间复杂度是 `O(n)`。
- **原地操作的空间复杂度是 O(1)**。

### 5. 案例分析：5 个例子建立直觉

下面用 5 段 C++ 代码把上面的规则跑一遍。每段代码都对应一个复杂度组合。

#### 示例 1：单层循环，时间 O(n) 空间 O(1)

```cpp
#include <bits/stdc++.h>
using namespace std;

// 求数组所有元素之和
int getSum(const vector<int>& nums) {
    int sum = 0;
    for (int i = 0; i < (int)nums.size(); i++) {
        sum += nums[i];
    }
    return sum;
}
```

- 时间：`O(n)`，n 是 `nums.size()`，for 循环跑 n 次。
- 空间：`O(1)`，只用了 `sum` 一个 int 变量。

#### 示例 2：分支结构不影响最坏时间复杂度

```cpp
#include <bits/stdc++.h>
using namespace std;

// n 是 10 的倍数时求累加和，否则返回 -1
int sumIfMultipleOfTen(int n) {
    if (n % 10 != 0) {
        return -1;
    }
    int sum = 0;
    for (int i = 0; i <= n; i++) {
        sum += i;
    }
    return sum;
}
```

- 复杂度分析**只看最坏情况**：`n = 10` 时确实只跑一次返回 -1，但是 `n = 1000` 也要跑 1000 次；最坏情况仍是 `O(n)`。
- 空间 `O(1)`，只用 `sum`。

#### 示例 3：嵌套循环，时间 O(n²) 空间 O(1)

```cpp
#include <bits/stdc++.h>
using namespace std;

// 数组中是否存在两个数之和等于 target
bool hasTargetSum(const vector<int>& nums, int target) {
    int n = (int)nums.size();
    for (int i = 0; i < n; i++) {
        for (int j = i + 1; j < n; j++) {
            if (nums[i] + nums[j] == target) {
                return true;
            }
        }
    }
    return false;
}
```

- 内层循环不是从 0 跑到 n，实际执行次数 < n²，但**我们分析的是上界** → `O(n²)`。
- 空间 `O(1)`，只用了 `i, j` 两个变量。

#### 示例 4：内存申请本身就带 O(n) 时间

```cpp
#include <bits/stdc++.h>
using namespace std;

// 申请一个长度为 n 的数组
vector<int> allocArray(int n) {
    return vector<int>(n, 0);
}
```

- 「`vector<int>(n, 0)`」要执行 n 次默认初始化（int 置 0），时间是 `O(n)`。
- 空间也是 `O(n)`，因为返回了一个 n 长度的数组。
- 这就是「**隐藏的时间复杂度**」：并不是 for 循环才耗时间，库函数本身也有时间开销。

> [!tip] 经验
> 标准库容器在创建时通常要初始化所有元素。如果你只想要一个「全 0 数组」做占位，复杂度是 O(n)；如果只是要「n 个槽位」，可以用 `vector<int>(n)`（默认 0 初始化）或者 `vector<int> v; v.reserve(n)`（不初始化，O(1)）。

#### 示例 5：返回新数组，时间 O(n) 空间 O(n)

```cpp
#include <bits/stdc++.h>
using namespace std;

// 把每个元素平方后返回新数组
vector<int> squareArray(const vector<int>& nums) {
    int n = (int)nums.size();
    vector<int> res(n);
    for (int i = 0; i < n; i++) {
        res[i] = nums[i] * nums[i];
    }
    return res;
}
```

- 时间：创建 `res` 占 `O(n)` + for 循环 `O(n)` = `O(n)`。
- 空间：返回的 `res` 是新数组，长度 n → `O(n)`。

### 6. 复杂度分析的「陷阱」清单

| 陷阱 | 说明 | 正确认识 |
| --- | --- | --- |
| `while (n > 1) n = n / 2;` 是 O(n)？ | 不对！循环次数 = `log₂ n` | `O(log n)` |
| `unordered_map.find` 是 O(1)？ | 平均是 O(1)，最坏是 O(n) | 表述时要说「平均 O(1)」 |
| `vector.push_back` 是 O(1)？ | 摊销是 O(1)，单次最坏 O(n)（扩容） | 表述时说「摊销 O(1)」 |
| `string s = str1 + str2;` 是 O(1)？ | 错，会拷贝整段 str1 | O(len(str1)) |
| 递归 `f(n) = f(n-1) + O(1)` | 看起来像 O(n)，但展开后是 n 次调用 | `O(n)`，但要会算「递归树」 |

### 7. 一份「实用」的复杂度速查表

> 完整版速查表在 [[39-sort-basic|排序总览]] 给出。下面是复杂度章节的精简版。

| 复杂度 | 常见场景 | n=1e6 时大概耗时（参考） |
| --- | --- | --- |
| O(1) | 哈希查找、数组按下标 | 几纳秒 |
| O(log n) | 二分查找、平衡 BST 查找 | 约 20 步 |
| O(n) | 单层 for 循环、链表遍历 | 几毫秒 |
| O(n log n) | 快排、归并、堆排 | 几十毫秒 |
| O(n²) | 冒泡、选择、插入 | 几小时 |
| O(2ⁿ) | 暴力枚举子集、斐波那契递归 | 天文数字 |

> [!warning] n=1e6 的 O(n²) 大约是 10¹² 次操作。CPU 单核按 1e9 ops/s 算要 ~1000 秒 ≈ 17 分钟，**基本就超时了**。所以 n=1e6 时只能承受 O(n log n) 或更优。

## 📊 复杂度一览

| 概念 | 关键结论 |
| --- | --- |
| Big O 含义 | 增长率上界，去掉常数和低阶项 |
| 时间复杂度估算 | 粗略看「嵌套 for 循环层数」 |
| 空间复杂度估算 | 算法**额外申请**的存储 |
| 最坏 vs 平均 | 笔试默认最坏；哈希表/堆会单独标注「平均」 |
| 摊销 | 单次可能贵，整体平均便宜（如 push_back） |

## 🛠️ 应用场景

- **面试**：几乎每道题都要先说「时间复杂度 X、空间复杂度 Y」再写代码。
- **笔试**：n=1e5 / 1e6 的输入规模下，能不能过取决于你选的是 O(n log n) 还是 O(n²)。
- **工程优化**：发现某段代码 O(n²) 拖慢整体，是常见的性能瓶颈定位起点。
- **算法学习前置**：学 [[39-sort-basic|排序]]、[[24-binary-tree-traverse-basic|二叉树遍历]]前必须搞懂 O 符号。
