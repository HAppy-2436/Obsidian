# 选择排序所面临的问题

## 学习目标

- 理解 **选择排序（Selection Sort）** 的核心思路和代码实现
- 通过 Big O 分析得出其 O(N²) 时间复杂度、O(1) 空间复杂度
- 认识到选择排序是 **不稳定排序**，并理解为什么
- 体会「**为什么选择排序无法被优化**」背后的思维方法

## 一句话总结

**选择排序** 是最简单朴素的排序算法：每趟从未排序部分找到最小值，与分界线元素交换。问题在于它 **时间复杂度高（O(N²)）、对初始有序度无感、且不是稳定排序**——所有基础排序的优化都从这里开始。

## 1. 思路：从「最直觉」出发的排序

如果你从来没学过排序，给你一个数组让你从小到大排，你会怎么想？

> 最直接的思路：先遍历一遍数组，找到最小值，与第一个元素交换；再遍历剩下的，找到次小值，与第二个元素交换；…… 循环往复，直到整个数组有序。

这就是 **选择排序（Selection Sort）**——每趟 **选择** 当前未排序部分的最小元素，放到分界线左侧。

## 2. 代码实现（C++）

```cpp
#include <bits/stdc++.h>
using namespace std;

namespace dsa {

// 选择排序（Selection Sort）
// 时间复杂度：O(N²)
// 空间复杂度：O(1)，原地排序
// 稳定性：❌ 不稳定
void selectionSort(vector<int>& nums) {
    int n = nums.size();
    // sortedIndex 是分割线：[0, sortedIndex) 已排序，[sortedIndex, n) 未排序
    int sortedIndex = 0;
    while (sortedIndex < n) {
        // 找到未排序部分 [sortedIndex, n) 中的最小值下标
        int minIndex = sortedIndex;
        for (int i = sortedIndex + 1; i < n; i++) {
            if (nums[i] < nums[minIndex]) {
                minIndex = i;
            }
        }
        // 交换最小值到分界线位置
        swap(nums[sortedIndex], nums[minIndex]);
        // 分界线右移一位
        sortedIndex++;
    }
}

} // namespace dsa

int main() {
    vector<int> nums = {3, 1, 4, 1, 5, 9, 2, 6};
    dsa::selectionSort(nums);
    for (int x : nums) cout << x << " ";   // 1 1 2 3 4 5 6 9
    cout << endl;
    return 0;
}
```

> [!warning] LeetCode 912 会超时
> 这个算法逻辑正确，但拿到 [[#912|力扣 912 题「排序数组」]] 提交会 **超时**。时间复杂度 O(N²) 在 N = 5×10⁴ 时已经无法满足题目要求。

## 3. 三大指标分析

### 3.1 是否原地排序？

✅ **是原地排序**。整个过程只用 `sortedIndex`、`minIndex`、`i` 几个变量，没有申请额外数组。空间复杂度 O(1)。

### 3.2 时间复杂度

算法本质是一个嵌套循环：

```text
for sortedIndex in [0, n):
    for i in [sortedIndex+1, n):
        比较 + 可能更新 minIndex
```

总比较次数是 `(n-1) + (n-2) + ... + 1 = n(n-1)/2 ≈ n²/2`，所以：

- **时间复杂度**：O(N²)
- **操作次数**：约 N²/2 次比较 + 最多 N 次交换

> [!warning] 时间复杂度与初始有序度无关
> 关键点：即便输入数组已经有序，选择排序依然要执行 n²/2 次比较，**初始数据的有序度对算法时间复杂度没有任何影响**。这是 [[40-bubble-sort|冒泡排序]] 的「加 flag 提前终止」要解决的问题之一。

### 3.3 排序稳定性

选择排序是 **不稳定排序**，原因是「**跨距离交换**」会改变相同元素的相对顺序。

```text
初始：[2, 2', 1]
       ^      ^
 sortedIndex  minIndex

第 1 趟交换后：[1, 2', 2]
                ^      ^
                ↑      ↑
            sortedIndex 已移到 1，原本的 2 被挤到了 2' 之后

最终结果中 2 在 2' 之前，相对顺序被破坏 → 不稳定
```

如果只是把最小值 **插入** 到 sortedIndex 位置（不交换 `nums[sortedIndex]` 到 `nums[minIndex]`），就能保持稳定——这是 [[40-bubble-sort|冒泡排序]] 的第一波优化思路。

## 4. 为什么选择排序「无法被优化」？

这个问题非常有意思，思路本身比答案更有价值。

### 4.1 一个失败的尝试：suffixMin 预计算

观察算法会发现：每次找最小值都从头遍历，存在 **冗余计算**。

能否在遍历 nums[0..] 时顺便算出 nums[1..]、nums[2..] 的最小值？

定义 `suffixMin[i] = min(nums[i..])`，这样后续每次找最小值就是 O(1) 查表，看起来就能把内层循环消掉，让时间复杂度变成 O(N)。

**答案是不行**。原因在于：每次交换都会改变原数组 nums，导致提前算好的 suffixMin **失效**。

### 4.2 思维价值

虽然这次尝试失败了，但背后的思路是经典的：

- **预计算 + 缓存**：通过额外空间换时间（如前缀和、后缀和）
- **失效检测**：当原数据动态变化时，预计算结果会失效（这类问题在线段树章节会详细解决）

> [!tip] 「为什么有些问题无法优化」本身是宝贵的认知
> 知道「**选择排序无法被简单优化**」之后，你就会明白后续所有 O(N²) 突破 O(N²) 的排序算法（希尔/快排/归并/堆排），都引入了 **全新的算法思想**——不是对选择排序的局部修补，而是重新设计问题分解方式。

## 6. 优缺点总结

**优点**：

- 实现最简单，代码清晰
- 交换次数少（最多 N 次），对于「交换代价高」的硬件（如写入 EEPROM）很友好
- 不依赖输入数据的有序度

**缺点**：

- 时间复杂度 O(N²)，大数据量完全不可用
- 不稳定，对于多 key 排序不友好
- 即便输入有序也跑满 n²/2 次比较
