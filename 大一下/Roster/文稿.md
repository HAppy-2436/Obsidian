最近刷到了一个有趣的小项目，可以统计电脑上的chromium内核数量，确实给我带来了一些震撼，他让我留意到了这个问题：
# 你电脑里到底有多少chromium
——电脑上到底有多少chromium。而我了解到了一个试图打破这个局面的新兴技术
# Electrobun
这就是我今天为大家介绍的——Electrobun，官方的描述是“用TypeScript构建极快、极小、跨平台的桌面应用”，这里的“极快”、“极小”便和我们熟悉的electron形成了鲜明的对比
# Star History
作为一个新兴的桌面应用框架，当前10k+的star数，以及26年以来关注度的飙升，都印证了他切实的在试图解决当下桌面应用开发的痛点
# 第一部分
那么他针对的问题，就是桌面应用开发背负的历史包袱。
# 为什么要重复造轮子
站在如今的视角审视，难以理解为什么桌面应用要各自打包一个浏览器内核，为什么要重复造轮子，但事实上，现在看起来的问题，是曾经问题的解决方案
# 2013年
时间回到13年前，那时主流的系统自带的webview功能都极其有限，支持现代的HTML5、CSS3都是天方夜谭，依赖系统webview开发桌面应用完全不可行。
# 唯一的出路
于是，开源、跨平台一致性优异、性能强悍且支持最新web标准的chromium成为了当时时代下的最优解
# Node.js
浏览器不具备的读写文件、调用系统API、创建窗口等等系统能力，Nodejs也填上了最后一片拼图，让 JavaScript 能够跳出浏览器，直接与操作系统进行交互
# Electron
二者结合，便诞生了统治桌面应用开发至今的Electron框架
# 应用
我们熟知的诸多现代跨平台桌面软件，都是基于此的。但是，强大泛用的背后，一个空白的应用安装包却都要占用150MB起步
# 系统webview
但2026年的今天，我们发现，当年那个迫使我们必须打包 Chromium 的核心矛盾，已经发生了变化。现在主流的操作系统自己，已经为我们准备好了现代、高效的 WebView。意味着我们不再需要自己‘随身携带’一个上百兆的 Chromium 了
# Bun
与此同时，在运行时层面，我们也迎来了一个新的选择。新秀Bun与Node.js相比，体积和启动时间上都有很大的优势，而且它开箱即用地支持TypeScript，内置了测试、打包等功能，大大简化了开发环境。
# Electrobun
已经成熟、0MB 额外成本的系统 WebView，加上是更轻、更快、仅需 10MB的 Bun 运行时，再加上typescript，我们便得到了安装包只有12MB，启动仅需0.5s的Electrobun，性能上得到飞跃。
# 第二部分
接下来我来为大家详细介绍Electrobun的四个核心特性
# 特性一
可以清晰的看到，和electron对比，新框架的webview占用直接从90MB清零，运行时的占用也只有Node的四分之一，这便是性能上极大的优势
# 运行内存
除了体积和启动速度之外，桌面应用还有一个老大难问题便是运行时的内存占用，每个Electron应用都有独立的Chromium进程，而且随着使用时间增加，内存泄漏问题会越来越严重。而得益于没有常驻的庞大 Chromium 引擎进程，加上底层是内存管理及其克制的zig语言实现的，一个典型应用，运行时的内存占用可以控制在40MB左右
# 特性二
接下来，我们来看第二个核心特性：类型化 RPC。面对主进程和渲染进程之间的通信，Electron里传统的IPC依赖字符串，让通信过程松散、不安全。无论是事件名还是参数类型，编译器都无法帮你检查。如果事件名拼写错了，或者参数类型不匹配，只有在运行时才会报错，严重拖慢开发效率。
而 Electrobun 提供了完全类型化的 RPC。在主进程中用 `.define` 方法定义一个函数，并明确声明参数和返回值类型。然后，在渲染进程中调用时，你就能获得完美的IDE 智能感知和编译时校验。如果参数写错了，编辑器会立刻给你红线警告。这种从运行时错误到编译时检查的转变，极大地提升了大型项目的开发体验和代码质量。
# 特性三
第三个特性，是关于**更新**的。假设你的应用发布了一个新版本，只改了一行代码里的一个错别字。用户需要更新，但传统 Electron 更新方式会要求用户下载150MB的完整安装包，非常不友好。
Electrobun 采用了bsdiff 字节级补丁技术。它能够精确地比对旧版本文件和新版本文件的二进制差异，只生成一个包含变化部分的、**极小**的补丁文件。比如这个例子，只改了一行字，补丁可能就只有14KB，可以实现用户无感更新。
# 补丁链
bsdiff 不仅高效，还有非常巧妙的应用方式，补丁链。面对间隔数个版本的更新，可以逐步下载每次的补丁，渐进式地升级到最新版，而不是被迫下载完整安装包。
在不同平台上electrobun也分别做了精心的适配，细节处理非常到位。
# 特性四
最后一个核心特性，是它的底层实现语言——**Zig**。这是一个新兴的系统级编程语言。zig编写的核心层处于上层运行时和底层平台原生实现之间。他有着一些独特的优势，它**没有运行时开销**，编译出的二进制文件极小；它采用**显式的内存管理**，没有垃圾回收，所以没有性能毛刺；它可以**无缝地调用 C 语言**，直接与操作系统 API 对话；最后，它拥有优秀的的交叉编译能力，使得为不同平台编译应用变得异常简单舒适。这为 Electrobun 的‘轻量’和‘快速’打下了坚实的基石。
# 竞品
选择抛弃Chromium实现轻量化的桌面框架不止这一个，还有例如用Rust作后端的Tauri和用Go的Wails。但是Electrobun让开发者可以**完全留在前端的舒适区**。你可以无缝复用 npm 上数百万的包，享受 Bun 提供的**毫秒级热更新**。它的学习曲线最低，能让一个普通的前端开发者在半小时内上手，构建一个真正的桌面应用。这就是它最大的优势。
# 缺陷
但任何技术选型，都是权衡与取舍。
选用Electrobun也有非常多的风险，例如抛弃了 Chromium，就意味着你把渲染权交给了用户的操作系统，丧失了渲染一致性，对于某些高级 CSS 特性或动画，不同平台的表现可能完全不同。
其次，是**底层生态**。Bun 虽然快，但毕竟是新秀。它对很多复杂的 Node.js 原生 C++ 插件，比如一些底层的 SQLite 或图像处理库 (Sharp) 的兼容性，目前还无法做到 100%。如果你的业务刚好踩到这些深水区，遇到报错，你很可能在网上找不到任何现成的解决方案，甚至需要自己去看 Zig 源码来解决问题。这对一些初创团队或工期紧张的项目来说，是一个不小的挑战。
架构选型永远是权衡。Electrobun 给你的，是‘用渲染不可控的小概率风险，来换取那下降 90% 的体积和内存占用’。对于大部分开发轻量级工具、内部应用或者注重分发效率的开发者来说，值得考虑
